# One-page section outline

**Introduction.** The introduction will frame live screen-share AI assistants as real-time multimodal systems that can observe screen and speech streams while helping with active tasks. It will name the consent-capability gap: users lack fine-grained runtime control over what the assistant may see, retain, and say. The contribution scope is a runtime mediation layer, positioned in the IUI/SOUPS direct-title lineage of Wijesekera and colleagues' dynamic permission work [Wijesekera 2017] and Liu and colleagues' Personalized Privacy Assistant [Liu 2016]. The section depends on the design choice that control is enforced between capture adapters, session memory, and assistant output rather than only in a static settings panel. It will not claim measured reduction, user acceptance, or deployment readiness until the synthetic metrics and any later user-study path exist.

**Problem and threat model.** This section will define the scoped hazards created by raw screen and speech access: over-broad observation, retention after consent changes, output disclosure, screen-visible prompt injection, and context drift during fast interface changes. The contribution scope is a threat model for runtime mediation in synthetic sessions, not a general security model for all assistant deployments. The section depends on design choices already present in the threat model: bounded adversaries, prohibited surfaces, explicit policy actions, and synthetic fixtures that mark sensitive units and benign task-critical units. It will not cover operating-system compromise, poisoned model weights, network attackers, malicious extensions, or formal confidentiality proofs. Any mitigation language must map to sensitive exposure rate, false block rate, policy action accuracy, indirect-reference rate, latency overhead, or audit-log completeness.

**System design.** This section will describe the mediation boundary between raw multimodal inputs and assistant context or output. The contribution scope is the architecture of screen capture normalization, speech-event handling, consent policy, redaction, session memory gating, output guarding, and audit recording as separable components. The section depends on the design choice that screen and speech content are treated as untrusted until policy and redaction decisions run. It will not claim universal detector coverage, production hardening, tamper evidence, or end-to-end assurance beyond the implemented synthetic paths. Each component description must say what it consumes, what it emits, and which evaluation surface could reveal failure.

**Policy and consent model.** This section will specify the runtime rule objects that decide what the assistant may observe, retain, and say. The contribution scope is compositional consent control: session defaults, per-window or per-region overrides, sensitive-token categories, modality-specific controls, retention choices, and response constraints. The section depends on a policy-engine design choice that turns fixture annotations and active consent state into explicit actions before model context is constructed. It will not present the policy language as complete, formally proven, legally sufficient, or portable to every domain. Evaluation claims must stay tied to policy action accuracy and downstream sensitive exposure or false block measurements.

**Implementation.** This section will report only implemented artifacts in the review pack: event schemas, fixture loading, policy evaluation, redaction checks, memory gating, output filtering, audit logging, tests, and the smoke path. The contribution scope is a minimal runnable prototype for synthetic evaluation, not a full assistant product. The section depends on traceability from each implementation claim to a module, fixture, test, smoke output, or future benchmark output. It will not describe planned modules as completed, and it will not imply integration with live screen capture, live speech capture, external assistants, or non-synthetic operational data. Any code-level limitation should be stated where it constrains evaluation validity.

**Evaluation.** This section will define a synthetic-only benchmark over terminal credentials, chat-style messages, browser personal data, spoken sensitive fragments, screen-visible prompt injection, dense interfaces, zoom changes, small fonts, fast window switching, mixed-sensitivity screens, obfuscated credentials, and split or spaced personal data. The contribution scope is measuring how the mediated path differs from an unmediated baseline under controlled fixtures. The section depends on predeclared metrics: sensitive exposure rate, false block rate, latency overhead, task completion, policy action accuracy, indirect-reference rate, and audit-log completeness. It will not report human-subject findings, field outcomes, final benchmark numbers, or superiority claims until generated files under `eval/results/` exist. It will separate smoke-test coverage from benchmark evidence so runnable code is not mistaken for measured results.

**Limitations and ethics.** This section will state the limits that reviewers must keep attached to every claim. The contribution scope is synthetic design and measurement evidence for a runtime mediation layer, not validation of real user behavior or real meeting practice. The section depends on the design choice to exclude non-synthetic screen captures, non-synthetic speech, production telemetry, secrets, identifying metadata, and public repository links from the review pack. It will not imply ethics clearance, recruitment approval, broad user benefit, or operational readiness. It will specify that any later human-subject study needs an explicit ethics path and that current results, once available, only cover the synthetic benchmark.

**Related work.** This section will place the paper against contextual integrity, permission assistants, LLM privacy benchmarks, privacy-conscious assistants, prompt-injection defenses, and tool-level agent controls. The contribution scope is the runtime screen-share mediation boundary: it borrows from Nissenbaum's contextual integrity framing [Nissenbaum], contrasts with Mireshghallah and colleagues' ICLR 2024 LLM privacy benchmark [Mireshghallah 2024], distinguishes itself from the DeepMind 2024 contextual-integrity assistant work [Ghalebikesabi 2024], and positions screen-visible prompt injection against CaMeL and AgentSentinel-style control surfaces [CaMeL; AgentSentinel]. The section depends on the design choice that this paper mediates observation, retention, and response during live assistance rather than only deciding whether information should be shared in a form or whether a tool call should execute. It will not claim priority, broad novelty, or stronger privacy/security properties than the synthetic metrics support. Every comparative statement must be backed by final bibliography entries before submission.
